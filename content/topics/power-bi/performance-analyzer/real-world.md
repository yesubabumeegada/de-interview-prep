# Performance Analyzer — Real-World Patterns

## Pattern 1: Diagnosing a Slow Executive Dashboard

**Scenario**: The CEO's dashboard takes 45 seconds to load. It has 12 visuals with complex DAX measures on a 50M-row DirectQuery fact table.

### Step 1: Run Performance Analyzer

```
View → Performance Analyzer → Start Recording → Refresh Visuals

Results:
────────────────────────────────────────────────────────
Revenue Trend Line Chart     → DAX: 12,400ms  ← Slowest
Top 10 Products Bar Chart    → DAX:  8,200ms
KPI Cards (×4)               → DAX:  3,100ms
Sales Matrix by Region       → DAX:  6,900ms
Filter Slicer (Date)         → DAX:    340ms
Filter Slicer (Region)       → DAX:    280ms
────────────────────────────────────────────────────────
Total page load: ~45 seconds
```

### Step 2: Copy the Slowest Query to DAX Studio

```
Revenue Trend Line Chart → Copy query → Paste into DAX Studio
```

```dax
-- The copied query (auto-generated by Power BI)
DEFINE
    VAR __DS0Core =
        SUMMARIZECOLUMNS(
            DimDate[YearMonth],
            "Revenue", IGNORE([Total Revenue]),
            "RevenueSPLY", IGNORE([Revenue SPLY]),
            "YoY%", IGNORE([YoY Growth %])
        )
EVALUATE
    __DS0Core
ORDER BY DimDate[YearMonth]
```

### Step 3: Server Timings Analysis

```
Total: 12,400ms
  FE: 11,200ms (90%)
  SE:  1,200ms (10%)
SE Queries: 24 (!!)
```

24 SE queries for a simple line chart means the measure generates many sub-queries.

### Step 4: Identify the Culprit Measure

```dax
-- Current YoY Growth % (problematic):
YoY Growth % =
VAR Current = SUM(FactSales[SalesAmount])
VAR Prior = CALCULATE(
    SUM(FactSales[SalesAmount]),
    SAMEPERIODLASTYEAR(DimDate[Date])
)
RETURN DIVIDE(Current - Prior, Prior, 0)

-- For 24 months on the line chart, this generates:
-- 24 queries for Current + 24 queries for Prior = 48 SE queries
-- Then FE does the ratio for each month
```

### Step 5: Optimization

```dax
-- Optimized: Use CALCULATETABLE to pre-fetch both periods in fewer SE queries
YoY Growth % (Optimized) =
VAR CurrentPeriod =
    SUMMARIZE(
        CALCULATETABLE(FactSales, ALLSELECTED(DimDate)),
        DimDate[YearMonth],
        "@Current", SUM(FactSales[SalesAmount])
    )
VAR PriorPeriod =
    SUMMARIZE(
        CALCULATETABLE(FactSales, SAMEPERIODLASTYEAR(DimDate[Date])),
        DimDate[YearMonth],
        "@Prior", SUM(FactSales[SalesAmount])
    )
-- Note: For DirectQuery, consider creating an aggregation table instead
-- The real fix for DirectQuery line charts is an aggregation table:
```

```dax
-- Best fix: Aggregation table (AggSalesByMonth) reduces all 12 time-series queries to ~2 SE queries
-- Configure agg table in Model view → Manage Aggregations
-- After agg table: DAX query time drops from 12,400ms to 180ms
```

## Pattern 2: Diagnosing Memory Pressure

**Scenario**: After a dataset refresh, all visuals are slow for the first 30 minutes, then become fast. Users are complaining about slow "cold start".

### VertiPaq Analyzer Investigation

```
DAX Studio → Advanced → VertiPaq Analyzer → Refresh

Table: FactSales (500MB)
  Column: TransactionGUID (380MB — 76% of table size!)
  Column: SalesAmount (12MB)
  Column: DateKey (8MB)
  ... 12 more columns: 100MB total
```

**Finding**: `TransactionGUID` is a UUID column with 50M distinct values (one per row). It compresses to nothing — essentially storing 50M raw strings. This wastes 380MB of the 1GB Premium capacity memory.

### Fix: Remove the High-Cardinality Column

```powerquery
// In Power Query: remove TransactionGUID before loading
let
    Source = Sql.Database("server", "db"),
    FactSales = Source{[Schema="dbo",Item="FactSales"]}[Data],
    DateFiltered = Table.SelectRows(FactSales,
        each [OrderDate] >= RangeStart and [OrderDate] < RangeEnd),
    // Remove TransactionGUID — never used in any report
    RemovedColumns = Table.RemoveColumns(DateFiltered, {"TransactionGUID"})
in
    RemovedColumns
```

**After**: Model size drops from 500MB to 120MB. Cache warm-up time drops from 30 minutes to ~3 minutes. Cold start performance dramatically improves.

## Pattern 3: Query Reduction for Interactive Dashboard

**Scenario**: A dashboard has 8 slicers (Date Range, Region, Product Category, Customer Tier, etc.). Every slicer click triggers 15 visual re-queries. Users experience freezing on every filter change.

### Measuring the Problem

```
All Queries trace in DAX Studio while clicking slicers:
→ Single slicer click triggers 15 separate DAX queries
→ Each query: 200-800ms
→ Total per interaction: 3-8 seconds
→ User experience: "Why is it so slow when I click anything?"
```

### Fix 1: Query Reduction — Slicer Apply Buttons

```
File → Options → Query Reduction:
  ✅ Add an Apply button to basic slicers
  ✅ Add an Apply button to dropdown slicers
```

Now slicers show an "Apply" button. Queries are deferred until the user clicks Apply — they can select multiple slicer values without triggering intermediate re-queries.

### Fix 2: Edit Interactions to Disable Unnecessary Cross-Filtering

```
Format → Edit Interactions
  → Click on each visual
  → For visuals that shouldn't respond to certain other visuals, click the "None" icon
  → Example: A header card showing company name should not be cross-filtered by any slicer
```

### Fix 3: Page-Level Filters Instead of Slicer-to-Slicer Cross-Filtering

```dax
-- Replace a slicer-driven filter with a page-level filter
-- Instead of having an "Active Products Only" slicer:
-- Add a page-level filter: DimProduct[Status] = "Active"
-- This is applied once at page load, not repeatedly on interaction
```

## Pattern 4: Comprehensive Performance Optimization Checklist

**Scenario**: Inheriting an existing slow report. Systematic approach to diagnose and fix.

### Phase 1: Measure (Performance Analyzer)

```
□ Run Performance Analyzer → Refresh Visuals
□ Record top 5 slowest visuals by DAX query time
□ Note any visual display time > 2 seconds
□ Count total visuals on page (flag if > 15)
```

### Phase 2: Diagnose (DAX Studio)

```
□ Copy each slow query → run in DAX Studio with Server Timings
□ Record FE%, SE%, SE query count for each
□ Run VertiPaq Analyzer → note large columns and high cardinality
□ Enable All Queries trace → identify repeated identical queries
```

### Phase 3: Fix (Systematic)

```
Model-level fixes (biggest impact):
□ Remove unused columns identified in VertiPaq Analyzer
□ Add aggregation table for slow DirectQuery visuals
□ Sort fact table by primary filter column in Power Query
□ Add indexes on source database (for DirectQuery)

DAX-level fixes (high FE% measures):
□ Replace FILTER(Table, condition) with column filter in CALCULATE
□ Replace LOOKUPVALUE inside iterators with RELATED
□ Add VAR to prevent repeated evaluation
□ Replace RANKX with calculated column if rank is static

Report-level fixes:
□ Enable Query Reduction → Apply buttons on slicers
□ Edit Interactions → disable unnecessary cross-filtering
□ Move rarely-used visuals to drill-through pages
□ Replace high-row-count tables with Top N filtered visuals
```

### Phase 4: Validate

```dax
-- Create a benchmark measure to track improvement
Benchmark Timestamp =
"Query ran at " & FORMAT(NOW(), "HH:MM:SS") & " — " &
TEXT([Data Latency (Hours)], "0") & " hours since refresh"
```

Compare Performance Analyzer times before and after each fix. Document results:

| Visual | Before | After | Reduction |
|---|---|---|---|
| Revenue Trend | 12,400ms | 180ms | 98.5% |
| Top Products | 8,200ms | 95ms | 98.8% |
| KPI Cards | 3,100ms | 420ms | 86.5% |
| Sales Matrix | 6,900ms | 340ms | 95.1% |
| **Total Page** | **~45s** | **~2s** | **95.6%** |
