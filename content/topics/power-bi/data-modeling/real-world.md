# Data Modeling — Real-World Patterns

## Pattern 1: Sales Analytics Star Schema with Role-Playing Dates

**Scenario**: A retail company tracks sales with both an order date and a ship date. Finance wants to analyze revenue by order date; logistics wants to analyze by ship date.

### Model Design

```
DimOrderDate (copy of calendar) ──────┐
                                      ├── FactSales
DimShipDate (copy of calendar) ───────┘
DimProduct ────────────────────────────┘
DimCustomer ───────────────────────────┘
DimStore ──────────────────────────────┘
```

### Power Query: Create Role-Playing Date Tables

```powerquery
// In Power Query, create DimOrderDate from a shared calendar query
let
    // Base calendar
    Source = List.Dates(#date(2020,1,1), 365*7, #duration(1,0,0,0)),
    #"To Table" = Table.FromList(Source, Splitter.SplitByNothing(), {"Date"}),
    #"Changed Type" = Table.TransformColumnTypes(#"To Table", {{"Date", type date}}),
    #"Added Year" = Table.AddColumn(#"Changed Type", "Year", each Date.Year([Date]), Int64.Type),
    #"Added Month" = Table.AddColumn(#"Added Year", "MonthNum", each Date.Month([Date]), Int64.Type),
    #"Added MonthName" = Table.AddColumn(#"Added Month", "MonthName", each Date.ToText([Date], "MMMM"), type text),
    #"Added Quarter" = Table.AddColumn(#"Added MonthName", "Quarter", each "Q" & Text.From(Date.QuarterOfYear([Date])), type text),
    #"Added DateKey" = Table.AddColumn(#"Added Quarter", "DateKey", each Date.Year([Date]) * 10000 + Date.Month([Date]) * 100 + Date.Day([Date]), Int64.Type)
in
    #"Added DateKey"

// DimShipDate references the same base query (avoids data duplication in Power Query)
// Both tables are identical in structure; named differently for clarity in the model
```

### DAX Measures

```dax
-- Active relationship uses OrderDate
Revenue by Order Date = SUM(FactSales[SalesAmount])

-- Inactive relationship: use USERELATIONSHIP
Revenue by Ship Date =
CALCULATE(
    SUM(FactSales[SalesAmount]),
    USERELATIONSHIP(FactSales[ShipDateKey], DimShipDate[DateKey])
)

-- Days to Ship average
Avg Days to Ship =
AVERAGEX(
    FactSales,
    FactSales[ShipDateKey] - FactSales[OrderDateKey]
)

-- Orders not yet shipped (ship date is blank)
Open Orders =
COUNTROWS(FILTER(FactSales, ISBLANK(FactSales[ShipDateKey])))
```

## Pattern 2: Budget vs Actuals with Differing Granularity

**Scenario**: Actuals are at daily grain per product per store. Budget is at monthly grain per product category per region. Need to compare actual vs budget.

### Model Design

```
DimDate ──── FactActualSales (daily × product × store)
DimDate ──── FactBudget (monthly × category × region)
DimProduct ── FactActualSales
DimCategory ─ FactBudget
DimStore ──── FactActualSales
DimRegion ─── FactBudget
```

**Key insight**: Do not join FactActualSales and FactBudget directly. Use the shared dimensions as bridges.

### DAX Measures

```dax
-- Actual at any grain
Total Actual Revenue = SUM(FactActualSales[SalesAmount])

-- Budget: aggregate monthly budget to whatever grain the report uses
Total Budget =
SUMX(
    VALUES(DimDate[YearMonth]),
    CALCULATE(
        SUM(FactBudget[BudgetAmount]),
        TREATAS(
            VALUES(DimDate[YearMonth]),
            FactBudget[YearMonth]
        )
    )
)

-- Variance
Variance vs Budget = [Total Actual Revenue] - [Total Budget]

Variance % =
DIVIDE([Variance vs Budget], [Total Budget], 0)

-- Traffic light measure for conditional formatting
Budget Status =
VAR pct = [Variance %]
RETURN
    SWITCH(
        TRUE(),
        pct >= 0.05, "Green",
        pct >= -0.05, "Amber",
        "Red"
    )
```

## Pattern 3: Dynamic What-If with Disconnected Parameter Tables

**Scenario**: Management wants to explore how revenue changes if prices increase by X% or if volume grows by Y%.

### Disconnected Parameter Tables

```dax
-- Create via "New Parameter" in Modeling tab (generates this automatically)
Price Increase % = GENERATESERIES(0, 0.5, 0.01)
// Creates table with column [Price Increase %] from 0 to 50% in 1% steps

Volume Growth % = GENERATESERIES(-0.2, 0.5, 0.01)
// Creates table with column [Volume Growth %] from -20% to +50%
```

Power BI auto-generates a slicer and a parameter measure for each What-If parameter:

```dax
-- Auto-generated by Power BI
Price Increase % Value =
SELECTEDVALUE('Price Increase %'[Price Increase %], 0)

Volume Growth % Value =
SELECTEDVALUE('Volume Growth %'[Volume Growth %], 0)
```

### Scenario Measures

```dax
-- Simulated Revenue
Simulated Revenue =
VAR BaseRevenue = [Total Revenue]
VAR PriceAdj = 1 + [Price Increase % Value]
VAR VolumeAdj = 1 + [Volume Growth % Value]
RETURN
    BaseRevenue * PriceAdj * VolumeAdj

-- Revenue Uplift from Parameters
Revenue Uplift =
[Simulated Revenue] - [Total Revenue]

-- Simulated Gross Margin (assuming costs don't scale with price increase)
Simulated Gross Margin =
VAR SimRev = [Simulated Revenue]
VAR ActualCOGS = SUM(FactSales[COGS])
VAR SimCOGS = ActualCOGS * (1 + [Volume Growth % Value])
RETURN
    SimRev - SimCOGS

Simulated GM % =
DIVIDE([Simulated Gross Margin], [Simulated Revenue], 0)
```

## Pattern 4: Composite Model for Near-Real-Time Dashboard

**Scenario**: Executive dashboard needs near-real-time sales from a live SQL Server database, combined with static budget and product master data from an Excel file.

### Architecture

```
SQL Server (DirectQuery)     Excel (Import)
├── FactSales (live)         ├── DimProduct (master data)
└── DimCustomer (large)      ├── FactBudget (monthly)
                             └── DimRegion (lookup)
```

### Storage Mode Configuration

```
-- In Model view (no code needed, but the logic):
FactSales     → DirectQuery  (real-time, large)
DimCustomer   → Dual         (works with both Import and DQ)
DimProduct    → Import       (static, from Excel)
FactBudget    → Import       (static)
DimRegion     → Import       (small lookup)
```

### DAX for Composite Model

```dax
-- This measure works across Import and DQ tables transparently
YTD Revenue vs Budget =
VAR ActualYTD =
    CALCULATE(
        SUM(FactSales[SalesAmount]),
        DATESYTD(DimDate[Date])
    )
VAR BudgetYTD =
    CALCULATE(
        SUM(FactBudget[BudgetAmount]),
        DATESYTD(DimDate[Date])
    )
RETURN
    ActualYTD - BudgetYTD

-- For DirectQuery tables, avoid complex iterators — prefer aggregation functions
-- This is efficient:
Live Revenue = SUM(FactSales[SalesAmount])

-- This may be slow against DirectQuery:
Slow Measure =
SUMX(
    FactSales,
    IF(FactSales[SalesAmount] > 100, FactSales[SalesAmount], 0)
)

-- Better — push the filter to the database:
Fast Measure =
CALCULATE(
    SUM(FactSales[SalesAmount]),
    FactSales[SalesAmount] > 100
)
```

### Query Folding Verification

For DirectQuery measures, verify the SQL sent to the source:

```
-- In DAX Studio, use Server Timings to see:
-- 1. Number of SE (Storage Engine) queries sent to SQL Server
-- 2. SQL generated for each query
-- 3. Which calculations stay in DAX vs fold to SQL

-- Target: Maximum folding to SQL Server
-- Avoid: FE (Formula Engine) calculations on DirectQuery data
```
