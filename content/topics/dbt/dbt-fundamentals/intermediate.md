# dbt Fundamentals — Intermediate Concepts

## dbt Packages

Install community packages from [hub.getdbt.com](https://hub.getdbt.com):

```yaml
# packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: 1.1.1
  - package: dbt-labs/audit_helper
    version: 0.9.0
  - package: calogica/dbt_expectations
    version: 0.10.0
  - git: "https://github.com/your-org/internal-dbt-package.git"
    revision: main
```

```bash
dbt deps   # Install packages
```

### dbt_utils — Most Used Macros

```sql
-- Generate a surrogate key
SELECT
    {{ dbt_utils.generate_surrogate_key(['order_id', 'product_id']) }} AS sk,
    order_id,
    product_id

-- Date spine (generate a row per day)
{{ dbt_utils.date_spine(
    datepart="day",
    start_date="cast('2020-01-01' as date)",
    end_date="cast('2025-12-31' as date)"
) }}

-- Pivot columns
{{ dbt_utils.pivot('status', ['pending','shipped','delivered']) }}

-- Union all relations
{{ dbt_utils.union_relations(relations=[ref('orders_2022'), ref('orders_2023')]) }}
```

## Documentation

Every model and column can have YAML docs:

```yaml
# models/staging/schema.yml
version: 2

models:
  - name: stg_orders
    description: "Cleaned and standardized orders from the raw Postgres source."
    columns:
      - name: order_id
        description: "Primary key. Unique identifier for each order."
        tests:
          - unique
          - not_null
      - name: status
        description: "Order status: pending, shipped, delivered, cancelled"
        tests:
          - accepted_values:
              values: ['pending', 'shipped', 'delivered', 'cancelled']
```

### Docs Blocks (reusable descriptions)

```markdown
<!-- models/docs.md -->
{% docs order_id %}
Unique identifier for each order. Generated by the source system (Postgres autoincrement).
Never reused after deletion. References customer_id in the customers table.
{% enddocs %}
```

```yaml
# Reference in schema.yml
columns:
  - name: order_id
    description: "{{ doc('order_id') }}"
```

## Node Selection — Advanced

```bash
# Run only modified models (with state)
dbt run --select state:modified

# Run models modified vs production
dbt run --select state:modified+ --state path/to/prod/manifest

# Run by file path
dbt run --select models/marts/core/

# Intersection: tag AND folder
dbt run --select "tag:daily,marts.*"

# Union: tag OR folder
dbt run --select "tag:daily marts.*"

# Exclude modified tests (useful in CI)
dbt test --exclude state:modified
```

## Exposures

Document downstream consumers of your dbt models:

```yaml
# models/exposures.yml
version: 2

exposures:
  - name: revenue_dashboard
    type: dashboard
    maturity: high
    url: https://app.powerbi.com/...
    description: "Executive revenue dashboard — refreshed daily at 6am"
    depends_on:
      - ref('fct_orders')
      - ref('dim_customers')
    owner:
      name: Analytics Team
      email: analytics@company.com
```

Exposures appear in the lineage graph, making it clear what downstream tools depend on each model.

## Incremental Model Logic

```sql
-- models/fct_events.sql
{{ config(
    materialized='incremental',
    unique_key='event_id',
    on_schema_change='sync_all_columns'
) }}

SELECT
    event_id,
    user_id,
    event_type,
    event_ts
FROM {{ source('app', 'events') }}

{% if is_incremental() %}
    -- Only process new/updated rows on incremental runs
    WHERE event_ts > (SELECT MAX(event_ts) FROM {{ this }})
{% endif %}
```

## Metadata and Artifacts

dbt generates JSON artifacts after each run:

| Artifact | Contents |
|---|---|
| `manifest.json` | Full project graph, model metadata |
| `run_results.json` | Execution results, timings, row counts |
| `catalog.json` | Column types and row counts from warehouse |
| `sources.json` | Source freshness check results |

```bash
# Check source freshness
dbt source freshness

# Upload state for slim CI
dbt ls --output json > manifest.json
```

## dbt Permissions Model

Best practice for Snowflake:

```sql
-- Transformer role writes models
GRANT USAGE ON DATABASE PROD_DB TO ROLE TRANSFORMER;
GRANT CREATE SCHEMA ON DATABASE PROD_DB TO ROLE TRANSFORMER;

-- Reporter role reads marts only
GRANT SELECT ON ALL TABLES IN SCHEMA PROD_DB.ANALYTICS TO ROLE REPORTER;
GRANT SELECT ON FUTURE TABLES IN SCHEMA PROD_DB.ANALYTICS TO ROLE REPORTER;
```

## Project Governance Patterns

```
models/
├── staging/          # 1:1 with source tables, light cleaning only
│   └── _sources.yml  # Source definitions
├── intermediate/     # Complex joins, reusable building blocks
│   └── _int_models.yml
└── marts/            # Business-domain organized, consumption-ready
    ├── core/         # Cross-domain (orders, customers)
    ├── finance/      # Finance-specific
    └── marketing/    # Marketing-specific
```

**Layer rules:**
- Staging only references `source()`
- Intermediate only references `ref('stg_...')` or `ref('int_...')`
- Marts only reference `ref('int_...')` or `ref('stg_...')`
- Never reference a mart from staging or intermediate
