# Data Warehouse Design — Fundamentals


## 🎯 Analogy

Think of data warehouse design like designing a city: star schema streets are straight and fast (denormalized), data vault neighborhoods are modular and extensible (append-only history), and the warehouse's transit hubs (Redshift/Snowflake clusters) must handle rush hour (end-of-day reporting) without gridlock.

---
## OLTP vs OLAP

| Property | OLTP | OLAP |
|---|---|---|
| Purpose | Day-to-day transactions | Analytics and reporting |
| Operations | INSERT, UPDATE, DELETE | SELECT (complex aggregations) |
| Query type | Simple (1-10 rows) | Complex (millions of rows) |
| Schema | Normalized (3NF) | Denormalized (star/snowflake) |
| Data size | GBs | TBs to PBs |
| Latency | Milliseconds | Seconds to minutes |
| Users | Applications | Analysts, BI tools |
| Examples | PostgreSQL, MySQL | Snowflake, BigQuery, Redshift |

## Star Schema

The most common DW design: one central fact table surrounded by dimension tables.

```sql
-- Fact table: numeric measurements (what happened)
CREATE TABLE fact_sales (
  sale_id       BIGINT PRIMARY KEY,
  date_key      INT REFERENCES dim_date(date_key),
  product_key   INT REFERENCES dim_product(product_key),
  customer_key  INT REFERENCES dim_customer(customer_key),
  store_key     INT REFERENCES dim_store(store_key),
  -- Measures (additive):
  quantity      INT,
  unit_price    DECIMAL(10,2),
  total_amount  DECIMAL(12,2),
  discount_pct  DECIMAL(5,2)
);

-- Dimension tables: descriptive context (who, what, where, when)
CREATE TABLE dim_date (
  date_key      INT PRIMARY KEY,   -- surrogate key: 20240115
  full_date     DATE,
  day_of_week   VARCHAR(10),
  week_number   INT,
  month_number  INT,
  month_name    VARCHAR(10),
  quarter       INT,
  year          INT,
  is_holiday    BOOLEAN,
  is_weekend    BOOLEAN
);

CREATE TABLE dim_product (
  product_key   INT PRIMARY KEY,   -- surrogate key
  product_id    VARCHAR(20),       -- natural key from source system
  product_name  VARCHAR(200),
  category      VARCHAR(50),
  subcategory   VARCHAR(50),
  brand         VARCHAR(100),
  unit_cost     DECIMAL(10,2),
  is_active     BOOLEAN
);

CREATE TABLE dim_customer (
  customer_key  INT PRIMARY KEY,
  customer_id   VARCHAR(20),
  customer_name VARCHAR(200),
  email         VARCHAR(200),
  region        VARCHAR(50),
  country       VARCHAR(50),
  segment       VARCHAR(50),   -- Bronze/Silver/Gold
  registration_date DATE
);
```

**Why star schema?**
- Simple: analysts understand it intuitively
- Fast: few joins (fact + 1-2 dims)
- BI-friendly: Tableau/Power BI auto-detect star schemas

## Surrogate Keys vs Natural Keys

```sql
-- Natural key: from the source system (e.g., product_id = "SKU-12345")
-- Surrogate key: generated by the DW (e.g., product_key = 4521)

-- Why surrogate keys?
-- 1. Source systems can reuse IDs (deleted product SKU-12345 replaced by new product)
-- 2. Enables SCD (slowly changing dimensions) — multiple rows per natural key
-- 3. Faster joins (integer vs string)
-- 4. Decouples DW from source system changes

-- Surrogate key generation:
-- Option 1: Sequence/identity column (database-assigned)
CREATE SEQUENCE seq_product_key START WITH 1 INCREMENT BY 1;

-- Option 2: Hash (reproducible, content-addressable)
SELECT MD5(CONCAT(product_id, source_system)) AS product_key
-- Good for: recreating DW without a key registry
```

## Slowly Changing Dimensions (SCD)

When dimension values change over time (customer moves, product changes category):

```sql
-- SCD Type 1: Overwrite (no history)
UPDATE dim_customer SET region = 'EU' WHERE customer_id = 'C1001';
-- Use: when history doesn't matter (fix typos, update phone numbers)

-- SCD Type 2: New row (full history)
-- Add: start_date, end_date, is_current flag
CREATE TABLE dim_customer_scd2 (
  customer_key  BIGINT PRIMARY KEY,   -- new surrogate key per version
  customer_id   VARCHAR(20),          -- same natural key across versions
  customer_name VARCHAR(200),
  region        VARCHAR(50),
  start_date    DATE NOT NULL,
  end_date      DATE,                 -- NULL = current record
  is_current    BOOLEAN DEFAULT TRUE
);

-- Customer moved from US to EU:
-- Old record: end_date = today - 1, is_current = FALSE
-- New record: start_date = today, end_date = NULL, is_current = TRUE, region = 'EU'

-- Query: sales by region, using the region AT TIME OF SALE
SELECT d.region, SUM(f.total_amount)
FROM fact_sales f
JOIN dim_customer_scd2 d ON f.customer_key = d.customer_key
-- No date filter needed: customer_key in fact_sales points to the right version!
GROUP BY d.region;

-- SCD Type 3: Add column (limited history)
ALTER TABLE dim_customer ADD COLUMN previous_region VARCHAR(50);
UPDATE dim_customer
SET previous_region = region, region = 'EU'
WHERE customer_id = 'C1001';
-- Use: only need to compare current vs immediately previous value
```

## ▶️ Try It Yourself

```sql
-- Data warehouse design decision tree
-- Q1: Query pattern? OLAP (aggregates) → columnar (Snowflake, Redshift, BigQuery)
-- Q2: Schema? Known dimensions → Star. Evolving → Data Vault. Simple → Wide/OBT
-- Q3: SCD? History needed → Type 2. Just current → Type 1.

-- Example: dimension with SCD Type 2 + surrogate key
CREATE TABLE dim_customer (
    customer_sk     BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id     VARCHAR(50) NOT NULL,  -- Natural key
    name            VARCHAR(200),
    tier            VARCHAR(50),           -- Bronze/Silver/Gold
    region          VARCHAR(50),
    valid_from      DATE NOT NULL DEFAULT CURRENT_DATE,
    valid_to        DATE NOT NULL DEFAULT '9999-12-31',
    is_current      BOOLEAN NOT NULL DEFAULT TRUE,
    dw_load_ts      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Current snapshot view (hides SCD complexity from analysts)
CREATE VIEW v_current_customers AS
SELECT * FROM dim_customer WHERE is_current = TRUE;
```

> **Run it:** Copy the snippet into a REPL or file — no external services needed for the basic example.

## Interview Tips

> **Tip 1:** "Why do we use star schema instead of normalized schema in a DW?" — Normalized (3NF) schema minimizes redundancy, great for OLTP writes. But analytics queries need to join many tables to get results — expensive on large data. Star schema denormalizes dimensions (trades some storage for query speed). Analysts write simple 2-3 table joins instead of 10+ table joins. BI tools like Tableau work better with star schemas (auto-detected hierarchies, drag-and-drop).

> **Tip 2:** "What is a fact table vs a dimension table?" — Fact table: stores measurable, numeric events (sales, clicks, transactions). Contains foreign keys to dimensions and numeric measures. Usually the largest table. Dimension table: stores descriptive attributes (product name, customer region, date details). Provides context for filtering, grouping, and labeling fact table data. Rule of thumb: if it answers "how much" or "how many" → fact. If it answers "who", "what", "where", "when" → dimension.

> **Tip 3:** "When would you use SCD Type 2 vs Type 1?" — Type 1 (overwrite): when historical accuracy doesn't matter. Examples: correcting a typo in a customer name, updating a phone number. Type 2 (new row): when you need point-in-time accuracy. Example: a customer moved regions — sales from when they were in US should still show as US region in historical reports. Type 2 enables "as-was" reporting. Type 1 only supports "as-is".
